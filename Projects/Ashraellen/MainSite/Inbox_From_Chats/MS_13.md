## 2. Accepted decisions

### 2.1. Clean pages are preferred over patches

Accepted site-development rule:

If a page edit through GitHub tools starts failing, getting blocked, becoming confused by stale versions, or requiring temporary workarounds, the preferred solution is to assemble a full clean `index.html` page and let the user upload it manually to the correct place in `main`, rather than adding JS/CSS patches.

Especially applies to:

- `index.html` pages;
- pages where meaning, navigation and structure matter;
- multilingual book/series pages;
- professional/fund-facing pages.

Patches may be used only as emergency temporary measures, and should later be removed or folded into the clean page.

### 2.2. Localized visible titles for Error 404 cycle

For the Error 404 / Ошибка 404 series, visible titles must be localized on every language page because titles carry meaning and are not merely labels.

Technical slugs may remain stable in English, but visible titles must be translated.

Polish accepted visible titles:

- Series: `Błąd 404: Boga nie znaleziono`
- Book I: `Dlaczego ja?`
- Book II: `Na wszystko Jego wola`
- Book III: currently `w przygotowaniu` until a meaningful title is approved.

Accepted technical path remains:

- `/pl/books/error-404-god-not-found/`
- `/pl/books/error-404-god-not-found/why-me/`

Reason: URL can be a stable technical/storage label, but the page facade must speak the local language.

## 6. Technical notes

### 6.1. Duplicate Professional page button debugging

The user reported a duplicate fifth button on both:

- `/ru/professional/`
- `/en/professional/`

The static HTML pages had already been manually cleaned and contained only four links in the PDF/professional documents section. The duplicate was not in the page HTML.

Cause identified:

- A leftover dynamic JavaScript patch in `assets/analytics.js` was adding an extra 12-month project proposal link after page load.
- The old patch looked for `/ru/professional/` or `/en/professional/`, found the `#pdf` section, and appended another proposal link.

Important operational lesson:

- Clean static page can still appear wrong if shared JS modifies it after loading.
- Avoid temporary shared-JS patches for page content.

### 6.2. Service worker/cache note

If corrected pages still show old behavior, possible cause is browser/service-worker cache. Earlier `analytics.js` registers `/sw.js`. If stale assets are served, check/bump `sw.js` cache or test with a cache-busting query.

### 6.3. Previous GitHub update warning

At one point, an attempted update of `assets/analytics.js` failed with a 409 SHA conflict, although an earlier response incorrectly implied success. Future work should verify GitHub write results before reporting a commit.

Rule:

- Never report commit success unless the tool returned a successful commit SHA.
- If GitHub returns `409` or SHA mismatch, refetch current file and retry with current SHA, or provide a clean manual file.
