## 1. Summary

This chat established the first working architecture and visual language for the Ashraellen main website.

The site is being built as one multilingual static website on GitHub Pages, with one threshold-style language entry and two primary modes:

- Research — observation, investigation, protocols of experience, texts, projects, archive.
- Public — voice, public materials, talks, posts.

The key conceptual formula accepted in the chat:

> One site. One axis. Two modes of presence.

The entrance pages are minimal, full-screen, cinematic and quiet. Internal pages use a constant upper-right header with `Ashraellen` and a small micro-menu. Each major section receives a specific background image and emotional temperature.

## 2. Accepted decisions

### 2.1 General site architecture

Accepted architecture:

- One website, not two separate sites.
- One domain / one repository / one deployment.
- Two modes inside one site:
  - Research.
  - Public.
- Main language entry pages act as a threshold, not a regular homepage.
- No ordinary global menu on the entry pages.
- Entry pages show only:
  - Ashraellen.
  - Three-line mode formula.
  - Two equal buttons/links: Research and Public.
  - language switcher.
  - small symbol / seal.
- The website should feel like a portal, not a marketing landing page.

### 2.2 Technical architecture

Accepted technical stack / approach:

- Static website on GitHub Pages.
- Repository used during this chat: `Ashraellen/ashraellen` for live site scaffolding.
- Main APM repository for memory and project documentation: `Ashraellen/apm`.
- Current GitHub Pages project URL:
  - `https://ashraellen.github.io/ashraellen/`
- Project Pages path issue was solved by computing base path dynamically:
  - GitHub Pages project path: `/ashraellen/`
  - future custom domain path: `/`
- Pages should use JavaScript base detection:

```html
<script>
(function(){
  const isGitHub = location.hostname.endsWith('github.io');
  const repo = location.pathname.split('/').filter(Boolean)[0];
  window.__BASE__ = (isGitHub && repo) ? `/${repo}/` : `/`;
})();
</script>
```

Then internal links are set in JS with `base + 'ru/research/'`, etc.

### 2.5 Logo / symbol decision

The Ashraellen symbol is not a decorative logo, but an identity sign received by the user in a vision.

Accepted use:

- Use as favicon.
- Use as small footer / seal symbol at the bottom of pages.
- Keep small, quiet, almost like a seal.
- Do not overuse or turn into a loud brand mark.
- For favicon, use only the symbol without the word `ASHRAELLEN`.
- Favicon generated files from RealFaviconGenerator are acceptable without renaming:
  - `android-chrome-192x192.png`
  - `android-chrome-512x512.png`
  - `apple-touch-icon.png`
  - `favicon-16x16.png`
  - `favicon-32x32.png`
  - `favicon.ico`
  - `site.webmanifest`
- Favicon paths on GitHub Project Pages can be tricky because `/ru/` pages look for relative files under `/ru/`; later page rewrites should fix favicon paths consistently.

### 2.6 Internal page header constant

Accepted internal page header pattern:

- On all internal pages except entry pages, put a simple constant top header.
- It is not fixed / sticky.
- It has no box, no border, no dark glass panel.
- It sits at the top of the page, right-aligned.
- It contains:
  - `Ashraellen`
  - micro-menu underneath, maximum three links.

Example for Russian Research:

```html
<header class="site-header">
  <p class="brand">Ashraellen</p>
  <div class="menu">
    <a id="toEntry" href="#">Вход</a> |
    <a id="toPublic" href="#">Публичное</a>
  </div>
</header>
```

For deeper pages, menu can be:

```html
<a id="toEntry" href="#">Вход</a> |
<a id="toPublic" href="#">Публичное</a> |
<a id="toResearch" href="#">Исследование</a>
```

## 6. Technical notes

### 6.1 GitHub Pages

The live site is currently a GitHub Project Pages site, not a user site.

Therefore:

- Correct base URL now:
  - `https://ashraellen.github.io/ashraellen/`
- `https://ashraellen.github.io/` shows GitHub Pages 404 unless a repository named `Ashraellen.github.io` exists.
- This is expected and not an error.

### 6.2 Redirect issue

The root redirect was originally wrong because it redirected to `/ru/` instead of `/ashraellen/ru/`.

Correct solution:

- Detect GitHub hostname with `location.hostname.endsWith('github.io')`.
- Detect repo segment from pathname.
- Use `window.__BASE__`.

### 6.3 Paths

Use dynamic link assignment for important internal links.

Example:

```js
const base = window.__BASE__ || '/';
document.getElementById('toEntry').href = base + 'ru/';
```

Avoid hardcoded root paths like `/ru/` while hosted as GitHub Project Pages.

### 6.4 CSS organization

Current accepted CSS split:

- `assets/style.css` — language entry pages.
- `assets/research.css` — Research main pages.
- `assets/public.css` — Public main pages.
- `assets/texts.css` — Research Texts page.
- `assets/talks.css` — Public Talks page.

Avoid generic CSS from other chats that redefines `.content`, `.badge`, etc. without following the established page structure.

### 6.5 Page layout convention for internal pages

Internal pages should follow:

```html
<div class="wrap">
  <div class="page">
    <header class="site-header">...</header>
    <main class="main">
      <div class="content">...</div>
    </main>
  </div>
</div>
```

For Talks, `.container` was used instead of `.content` to avoid collision with older styles.

### 6.6 Favicon

Favicon files exist / were discussed.

Need later:

- unify favicon link handling in all pages;
- ensure no leading `/` paths break GitHub Project Pages;
- optionally fix `site.webmanifest` paths if they include leading `/`.

### 6.7 Image filenames


- independent artistic research;
