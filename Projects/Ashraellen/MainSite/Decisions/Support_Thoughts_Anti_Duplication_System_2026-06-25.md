# MainSite — система антидублирования опорных мыслей

Дата фиксации: 2026-06-25  
Проект: Ashraellen.com / MainSite / Public Thoughts  
Статус: active rule

## Короткое правило

Перед созданием новой дуги опорных мыслей нельзя назначать новые номера и slug-и, пока кандидатные тексты не сверены с реестром уже опубликованных мыслей.

Основной рабочий файл:

```text
Projects/Ashraellen/MainSite/Content/Support_Thoughts_Source_Register.md
```

Дополнительный рабочий файл для просмотра Telegram:

```text
Projects/Ashraellen/MainSite/Content/Telegram_Channel_Scan_Log.md
```

## Главная цель

Цель системы — не найти все посты, которые ещё не использованы, а не допустить повторного использования уже раскрытой мысли под новым номером, названием или образом.

Повтором считается не только буквальный дубль, но и смысловое совпадение.

## Что считается дублем

### 1. Прямой дубль

Тот же текст, та же формула, тот же Telegram-пост или почти тот же фрагмент.

### 2. Формульный дубль

Другая обёртка, но та же ударная формула или почти тот же финальный вывод.

Пример проверки:

```text
Проблема не уничтожена. Но её пафос — да.
```

Нельзя потом выпускать ту же мысль как:

```text
Проблема осталась, но потеряла величие.
```

если это не новая самостоятельная разработка, а тот же смысловой узел.

### 3. Образный дубль

Тот же центральный образ используется как главный смысловой двигатель:

```text
пустой стул
грязная чашка
закрыть книгу
образ вместо человека
корона проблемы
```

Образ можно упоминать вторично, но нельзя снова делать его центром новой опорной мысли.

### 4. Тематический дубль

Новая формулировка раскрывает уже опубликованную тему без существенного нового поворота.

Пример: если уже есть мысль о том, что человек страдает не от факта, а от додуманного сериала, то новая мысль о том же механизме должна иметь другой предметный центр, другой конфликт и другой вывод. Иначе это дубль.

### 5. Финальный дубль

Текст идёт по другому пути, но приходит к тому же финальному удару. Такой текст можно использовать только как материал для расширения существующей мысли, а не как новую опорную мысль.

## Рабочий порядок перед новой дугой

1. Открыть `Support_Thoughts_Source_Register.md`.
2. Просмотреть уже опубликованные мысли по полям:
   - номер;
   - название;
   - формула;
   - тема;
   - центральный образ;
   - финальный удар;
   - slug.
3. Просмотреть ближайшие доступные посты Telegram-канала через публичное веб-представление.
4. Отобрать 10–12 кандидатов.
5. Для каждого кандидата проверить:
   - не совпадает ли формула;
   - не совпадает ли центральный образ;
   - не совпадает ли финальный удар;
   - не раскрыта ли уже эта же тема;
   - не был ли этот Telegram-пост уже использован.
6. Только после проверки выбрать 6 мыслей для новой дуги.
7. Присвоить номера 0019–0024, 0025–0030 и так далее.
8. После создания дуги обновить `Support_Thoughts_Source_Register.md` и `Telegram_Channel_Scan_Log.md`.

## Статусы в реестре

Использовать короткие статусы:

```text
done      — мысль опубликована на сайте
selected  — выбрана для следующей дуги, но ещё не опубликована
candidate — найден кандидат, ждёт проверки
skip      — не подходит для опорной мысли
merged    — использовано частично внутри другой мысли
duplicate — не использовать: совпадает с уже опубликованной мыслью
review    — требуется ручное решение пользователя
```

## Как работать с Telegram

Полный ZIP-экспорт канала не является основным рабочим инструментом. Он может лежать как архивный запас.

Основной рабочий способ:

1. Смотреть публичный канал через веб-представление Telegram.
2. Идти ближайшими доступными слоями, не пытаясь сразу каталогизировать весь канал.
3. После каждой рабочей сессии фиксировать последнюю просмотренную точку в `Telegram_Channel_Scan_Log.md`.
4. Если Telegram не отдаёт старые посты удобно, тогда использовать экспорт или прямые ссылки на посты.

## Что фиксировать после каждой дуги

Для каждой опубликованной мысли фиксировать:

```text
номер;
статус;
дуга;
русское название;
формула;
центральный образ;
тема;
slug;
asset;
source note;
duplicate guard note.
```

Если точная ссылка на Telegram-пост не была сохранена во время отбора, писать честно:

```text
source note: Telegram public channel, exact message link to be backfilled.
```

Это лучше, чем выдумывать ссылку.

## Главное ограничение

Не полагаться только на память чата. Антидублирование должно опираться на письменный реестр в APM.

Память модели может помочь заметить похожесть, но окончательное рабочее основание — `Support_Thoughts_Source_Register.md`.
