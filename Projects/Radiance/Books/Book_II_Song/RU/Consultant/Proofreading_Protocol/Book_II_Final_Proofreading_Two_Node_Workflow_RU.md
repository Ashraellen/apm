# Book II «Песня» — двухузловая схема финальной вычитки

Версия: 1.0  
Статус: рабочая операционная схема  
Назначение: зафиксировать финальный порядок вычитки Book II «Песня» без участия архитектурного / канонического чата как обязательного рабочего узла.

---

## 1. Главная схема

Финальная вычитка Book II работает по двухузловой модели:

```text
Чат-вычитчик → Контролёр вычитки
```

Этот архитектурный чат не участвует в регулярном цикле вычитки.

Он остаётся только архивно-стратегическим резервом для исключительных случаев:

1. серьёзный спор о каноне;
2. поломка архитектуры;
3. необходимость создать новую большую инструкцию;
4. решение автора вернуться к стратегическому уровню.

В обычной работе его не использовать.

---

## 2. Роли

### 2.1. Чат-вычитчик

Чат-вычитчик:

1. берёт официальный Draft;
2. читает его полностью;
3. сверяется с картой рисков;
4. делает минимальную бережную вычитку;
5. создаёт Final Markdown;
6. обновляет FINAL_INDEX;
7. даёт короткий отчёт;
8. передаёт файл контролёру;
9. после approve создаёт DOCX, если это требуется.

Главная формула вычитчика:

```text
Убрать шум, не тронув дыхание.
```

---

### 2.2. Контролёр вычитки

Контролёр вычитки становится финальным принимающим узлом после вычитчика.

Он:

1. сравнивает Draft и Final Markdown;
2. проверяет функцию фрагмента;
3. проверяет карту рисков;
4. проверяет метаследы;
5. проверяет персонажей;
6. проверяет предметность;
7. проверяет ритм;
8. проверяет финал;
9. проверяет, не стал ли текст слишком правильным;
10. выдаёт вердикт;
11. разрешает или не разрешает переход к DOCX.

Главная формула контролёра:

```text
Защитить форму, не переписывая текст.
```

---

### 2.3. Автор

Автор принимает стратегические решения.

Автор может:

1. запускать следующую главу;
2. передавать файлы между чатами;
3. принимать решение в спорных случаях;
4. остановить процесс;
5. запросить новую стратегическую инструкцию;
6. вернуть вопрос в архитектурный чат при необходимости.

Автор не обязан становиться техническим контролёром каждой мелкой правки.

---

## 3. Что убрано из схемы

Из регулярной схемы убран третий операционный узел:

```text
Главный канонический чат / архитектурный чат
```

Теперь не используется схема:

```text
Вычитчик → Контролёр → Главный чат
```

Используется схема:

```text
Вычитчик → Контролёр
```

Контролёр получает полномочия финального принимающего узла в пределах финальной вычитки.

---

## 4. Что передано контролёру

Контролёру переданы функции:

1. финальная проверка функции главы;
2. проверка канона Book II;
3. проверка карты рисков;
4. утверждение или возврат вычитанной версии;
5. проверка обязательных микроправок;
6. решение о переходе к DOCX;
7. фиксация итогового вердикта.

Контролёр не получает права переписывать художественный текст.

---

## 5. Рабочий цикл одной главы

```text
1. Автор даёт вычитчику команду на главу.
2. Вычитчик читает Draft.
3. Вычитчик открывает карту рисков.
4. Вычитчик делает минимальную вычитку.
5. Вычитчик создаёт Final Markdown.
6. Вычитчик обновляет FINAL_INDEX.
7. Вычитчик сообщает автору пути и commit SHA.
8. Автор передаёт Final Markdown контролёру.
9. Контролёр сравнивает Draft и Final.
10. Контролёр выдаёт verdict.
11. При approve вычитчик создаёт DOCX, если требуется.
12. FINAL_INDEX обновляется.
13. Процесс переходит к следующей главе.
```

---

## 6. Вердикты контролёра

Использовать только:

```text
approve
approve after micro-corrections
returned for another pass
reject / wrong file
```

Нет отдельного статуса:

```text
awaiting main canonical chat
```

Такой статус не используется.

---

## 7. FINAL_INDEX

В FINAL_INDEX не должно быть колонки или обязательного этапа:

```text
Main canonical chat approval
```

Достаточно фиксировать:

1. Draft path;
2. Final Markdown path;
3. Proofreader status;
4. Controller verdict;
5. Mandatory micro-corrections;
6. Risk tags;
7. DOCX status;
8. Notes.

---

## 8. Когда можно возвращаться в архитектурный чат

Только если:

1. контролёр и вычитчик расходятся по принципиальному вопросу;
2. нужно изменить сам протокол;
3. обнаружена крупная каноническая проблема;
4. текст требует не вычитки, а новой редакторской стратегии;
5. автор сам решает поднять вопрос выше.

Во всех обычных случаях не возвращаться.

---

## 9. Итоговая формула новой схемы

```text
Вычитчик убирает шум.
Контролёр защищает форму.
Автор принимает стратегические решения.
```

Никакого третьего обязательного чата между контролёром и автором нет.
